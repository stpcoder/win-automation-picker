# 단계별 점검표

오류 원인이 분명하지 않을 때 위에서 아래로 확인합니다. 처음 발견한 이상 항목에서 멈추고 해당 설정을 바로잡은 뒤 다시 시작합니다.

## 1. 관리자 PC

- [ ] 최신 `AEWorkbench.exe`를 실행했다.
- [ ] 설정 파일이 실행 파일과 같은 폴더에 있다.
- [ ] 통신 서버 `연결 확인`이 성공한다.
- [ ] 전용 서버 폴더가 다른 업무 폴더와 겹치지 않는다.
- [ ] 대상 실장기 PC의 최근 연락 시각이 허용 범위 안이다.

## 2. 설치 구조

- [ ] TFT/UTF 이름이 실제 Rack 이름과 같다.
- [ ] `TFT30-1` 같은 실장기 PC 이름이 중복되지 않는다.
- [ ] 각 실장기 PC에 실제 실장기 최대 네 대만 등록했다.
- [ ] 실장기 번호가 실제 표시와 같다.
- [ ] SoC, Binary, DRAM, Lot, 장착 자재 ID와 고장 상태가 입력되어 있다.

## 3. 실장기 PC

- [ ] 전달받은 폴더 이름과 이 PC 이름이 같다.
- [ ] `통신 시작` 뒤 `통신 중: PC 이름`이 표시된다.
- [ ] Windows 시간과 관리자 PC 시간이 크게 다르지 않다.
- [ ] 디스크에 로그와 화면을 저장할 여유가 있다.
- [ ] 절전 또는 화면 잠금 정책이 자동 조작을 막지 않는다.

## 4. SK Commander

- [ ] 실제 테스트 때와 같은 창 수, 위치, 크기와 화면 배율이다.
- [ ] 각 창 안에서 실장기 번호가 보인다.
- [ ] 실장기 번호, SoC, 자재, 테스트 이름·상태, 부팅 단계가 연결되어 있다.
- [ ] `대상 확인`에서 올바른 창과 항목이 표시된다.
- [ ] 텍스트·색상 상태 블록을 각각 시험했다.

## 5. SEQ와 자동 실행 순서

- [ ] SEQ 검사가 PASS다.
- [ ] `;` 뒤 공백, Grid 누락과 명령 순서를 확인했다.
- [ ] 자동 실행 순서 구성 검사가 PASS다.
- [ ] 실장기별 입력값이 비어 있지 않다.
- [ ] 같은 프로그램 창은 `${channel}` 조건으로 구별된다.
- [ ] 현재 PC에서 핵심 블록과 전체 순서를 시험했다.

## 6. 직접 COM

- [ ] SK Commander와 다른 터미널이 대상 COM을 사용하지 않는다.
- [ ] COM 대조에서 HWID 또는 USB Serial이 일치한다.
- [ ] Baud rate와 시리얼 형식이 실제 장비와 같다.
- [ ] Enter 또는 안전한 상태 조회 명령에 응답한다.
- [ ] 부팅 단계 문구가 예상과 같다.

## 7. Binary

- [ ] 승인된 다운로드 도구 경로와 버전이다.
- [ ] 대상 SoC와 Binary 정보의 SoC가 같다.
- [ ] XML과 관련 파일의 확인값이 맞다.
- [ ] Download/EDL 또는 ADB 식별값이 대상 실장기와 같다.
- [ ] 물리 스위치 또는 preloader 전환 절차를 확인했다.
- [ ] Format이나 제한 영역 쓰기의 별도 확인을 완료했다.

## 8. 상태와 결과

- [ ] 새로고침 뒤 최근 불러온 시각이 갱신됐다.
- [ ] 실행 ID가 지금 시작한 테스트와 같다.
- [ ] 실장기 번호 조건과 상태 조건이 같은 창을 가리킨다.
- [ ] Grid 진행 수와 최근 로그가 함께 갱신된다.
- [ ] 화면은 새 요청 ID로 받은 최신 파일이다.

점검표를 모두 통과했는데 문제가 반복되면 [지원 범위와 현장 확인](support-scope.md)의 문제 보고 항목을 준비합니다.
