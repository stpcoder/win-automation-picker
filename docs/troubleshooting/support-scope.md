# 지원 범위와 현장 확인

프로그램이 자동으로 검사할 수 있는 부분과 실제 장비에서 확인해야 하는 부분을 구분합니다. 화면에 버튼이 있다고 해서 모든 SoC와 다운로드 프로그램의 물리 절차가 자동으로 보장되는 것은 아닙니다.

## 프로그램에서 검사하는 항목

- 관리자 PC, TFT/UTF, 실장기 PC, 실장기의 연결 구조
- 실장기 PC 한 대당 실장기 최대 네 대 제한
- COM, HWID, USB Serial과 등록 정보 비교
- SEQ 기본 문법, Grid와 명령 구조
- 자동 실행 순서의 빈 대상, 빈 반복, 잘못된 조건
- 실장기별 입력값 누락
- 통신 요청, 중단 요청, 결과와 화면 요청의 구분
- Binary XML과 관련 파일의 확인값
- 등록 SoC와 Binary 정보의 SoC 일치 여부
- Download/EDL 또는 ADB 식별값 일치 여부
- 위험한 방식의 추가 확인 단계
- 로그·화면 보관 개수와 파일 크기 제한

## Windows 실장기 PC에서 확인할 항목

- SK Commander 실제 항목을 정확히 선택하는지
- Windows UI 항목 정보가 프로그램에서 읽히는지
- 창이 네 개일 때 실장기 번호 조건으로 정확히 구별되는지
- 사내 보안 프로그램이 입력 기록과 화면 캡처를 막지 않는지
- 실제 COM과 Baud rate에서 명령·응답이 안정적인지
- USB를 다시 연결했을 때 HWID 또는 Serial로 같은 실장기를 찾는지
- 관리자 권한 또는 실행 정책이 필요한지
- 절전, 화면 잠금, 원격 접속 종료 뒤에도 프로그램이 필요한 상태로 남는지

## 실장기와 다운로드 도구에서 확인할 항목

- Qualcomm 물리 스위치와 Download/EDL 진입 절차
- MediaTek preloader에서 필요한 `exit` 횟수, 간격과 준비 문구
- `Format + Download`의 실제 영향과 복구 절차
- SoC별 XML 선택 규칙과 다운로드 프로그램 인자
- BROM 또는 제한 영역 작업의 주소, 길이, 저장 위치
- 성공·실패를 나타내는 실제 종료 코드와 로그 문구
- 다운로드 완료 뒤 부팅 단계와 ADB 재인식
- 현장 승인 버전과 실행 파일 확인값

이 값은 제조사, SoC, 다운로드 프로그램 버전에 따라 달라집니다. 확인되지 않은 명령을 추정해 등록하지 않습니다.

## 테스트별로 확인할 항목

- SEQ 이름 규칙
- bootloader, 2nd bootloader, LK, OS 이동 순서
- 필요한 `exit`, clock, 온도, VDD 명령
- 4-Corner의 온도·전압 조합과 순서
- Grid 이름과 Grid별 로그 저장 형식
- PASS·FAIL 텍스트 또는 색상 규칙
- CBT/CA, DQ READ, DQ WRITE 마진의 측정 항목과 실제 단위

## 기능별 준비 수준

| 기능 | 프로그램에서 가능한 작업 | 현장 확인 없이 실행하면 안 되는 부분 |
|---|---|---|
| SK Commander 자동 조작 | 항목 선택, 클릭·입력 녹화, 반복·조건, Python 내보내기 | 실제 SK Commander 버전의 항목 인식과 창 구별 |
| 상태 확인 | 텍스트·색상 조건, AND·OR, 탭별 표시, 화면 요청 | 실제 상태 색상과 텍스트의 의미 |
| 직접 COM | 최대 네 대 연결, 키·문자 전송, SEQ 실행, Grid 로그 | 실제 Baud rate, 부팅 프롬프트와 명령 순서 |
| Binary | 외부 도구 등록, XML 검사, 장치 식별, 실행 기록 | 제조사 도구 명령, 물리 스위치, Format 영향 |
| DRAM 마진 | SoC·DRAM 설정 초안, 승인값, 실행 설정과 로그 구조 | SoC별 레지스터, 변환식, 허용 범위와 승인 기준 |

## 현장 확인 권장 순서

1. 사용하지 않는 실장기 한 대를 정합니다.
2. 읽기 전용 확인부터 시작합니다.
3. SK Commander 항목 선택과 상태 판정을 확인합니다.
4. 직접 COM은 Enter와 상태 조회 명령부터 확인합니다.
5. 일반 Binary 업데이트를 승인된 샘플로 확인합니다.
6. Format 또는 제한 영역 쓰기는 별도 승인 뒤 확인합니다.
7. 한 대의 결과와 로그가 맞은 뒤 같은 SoC의 다른 실장기로 넓힙니다.
8. 확인한 다운로드 도구 버전, Binary 정보와 절차를 기록합니다.

## 문제 보고에 포함할 정보

- 프로그램 버전과 Windows 버전
- TFT/UTF, 실장기 PC, 실장기 번호
- SoC, Binary 이름·버전, DRAM Part와 장착 자재 ID
- 사용한 SEQ와 자동 실행 순서 파일 이름
- 오류가 발생한 버튼 또는 동작 블록 이름
- 최근 상태 시각과 실행 ID
- 필요한 범위의 로그와 요청한 화면
- 기대한 결과와 실제 결과

계정 비밀번호, 사내 서버 비밀값, 개인 식별 정보는 보고 파일에 넣지 않습니다.
